---
title: Search Memories
api: GET https://api.remem.online/memories/search
description: Semantic search — returns the most relevant memories for a query.
---

## Overview

Search memories using natural language. You do not need to match exact words — Remem understands meaning. Ask "where does this user live?" and it finds "User is based in Lagos, Nigeria" even though none of the words overlap.

Results are ranked by hybrid scoring: **70% semantic relevance + 20% recency + 10% importance.** Every result includes `score_detail` showing exactly how it scored.

---

## Request

### Headers

| Name | Required | Value |
|------|----------|-------|
| `X-API-Key` | Yes | Your Remem API key |

### Query Parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `query` | string | Yes | — | Natural language search query. |
| `user_id` | string | Yes | — | Search within memories for this user. |
| `agent_id` | string | Yes | — | Search within memories for this agent. |
| `top_k` | integer | No | `5` | Number of results to return. Min 1, max 20. |
| `min_score` | float | No | `0.70` | Minimum hybrid score threshold. Use `0.0` to return everything. |
| `memory_type` | string | No | `null` | Filter by `episodic`, `semantic`, or `summary`. Omit to search all types. |

memories = client.recall(

### Memory Object Fields

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | Memory UUID |
| `content` | string | The stored memory text |
| `user_id` | string | User this memory belongs to |
| `agent_id` | string | Agent that stored this memory |
| `memory_type` | string | `episodic`, `semantic`, or `summary` |
| `importance` | float | Importance score 0.0–1.0 |
| `score` | float | Final hybrid score for this query |
| `score_detail` | object | Breakdown of how the score was calculated |
| `metadata` | object | Any extra data stored with the memory |
| `created_at` | string | ISO timestamp when memory was stored |
| `last_accessed` | string | ISO timestamp of last retrieval |
| `expires_at` | string or null | ISO timestamp of expiry. `null` means permanent |

## Error Responses

| Code | Meaning | Fix |
|------|---------|-----|
| `401` | Invalid or missing API key | Check your key |
| `422` | Missing required parameter | Include `query`, `user_id`, and `agent_id` |

## Notes

<AccordionGroup>
  <Accordion title="Understanding score_detail">
    Use `score_detail` to debug why a memory ranked where it did:

    ```python
    for m in memories:
        print(f"Content:    {m.content}")
        print(f"Cosine:     {m.score_detail['cosine']:.3f}")
        print(f"Recency:    {m.score_detail['recency']:.3f}")
        print(f"Importance: {m.score_detail['importance']:.3f}")
        print(f"Final:      {m.score_detail['final']:.3f}")
        print()
    ```

    If the wrong memory is ranking first, check which signal is pulling it up.
    High recency but low cosine means it is recent but not very relevant.
    High cosine but low recency means it is relevant but stale.
  </Accordion>
  <Accordion title="Tuning min_score">
    `0.70` (default) — balanced, works well for most use cases.
    `0.85+` — strict, only highly relevant memories.
    `0.50` — loose, returns anything remotely related.
    `0.0` — everything, useful for debugging empty results.

    Start with `0.0` when debugging. If you are getting empty results
    it usually means `min_score` is too high, not that the memories are missing.
  </Accordion>
  <Accordion title="Getting empty results">
    If search returns zero memories, check in this order:

    1. Set `min_score=0.0` — if results appear, your threshold is too high
    2. Check `user_id` and `agent_id` match exactly what you used when storing
    3. Call `GET /memories` to list all stored memories and confirm they exist
    4. Check that memories have not expired via `expires_at`
  </Accordion>
</AccordionGroup>
